# pgvector Backend

The `ancora-memory` crate ships a pgvector SQL generation layer under
`backends::pgvector`. It produces parameterised SQL for every operation the
`VectorStore` trait requires, without requiring a live database during tests.

## Prerequisites

1. PostgreSQL >= 14
2. `pgvector` extension installed: `CREATE EXTENSION IF NOT EXISTS vector;`
3. Enable the feature flag in your `Cargo.toml`:

```toml
ancora-memory = { version = "*", features = ["pgvector"] }
```

## Quick start

```rust
use ancora_memory::backends::pgvector_migration::full_setup_sql;
use ancora_memory::backends::pgvector::{
    upsert_sql, cosine_query_sql, hybrid_query_sql,
    serialize_payload, encode_vector,
};

// Generate DDL for a new collection
let ddl = full_setup_sql("documents", 1536, 16, 100).unwrap();
// Execute ddl against your Postgres connection...

// Encode and upsert a document
let payload_json = serialize_payload(&my_payload).unwrap();
let embedding_literal = encode_vector(&my_embedding);
let sql = upsert_sql("documents");
// Execute sql with (id, embedding_literal, payload_json)...

// Retrieve nearest neighbours
let query_sql = cosine_query_sql("documents", 10, 0);
// Execute query_sql with (embedding_literal,)...

// Hybrid dense + keyword query
let hybrid_sql = hybrid_query_sql("documents", 10, 0.7);
// Execute hybrid_sql with (embedding_literal, "search terms")...
```

## Indexing

### HNSW (default)

Best recall-per-latency for most workloads. Use for production.

```rust
use ancora_memory::backends::pgvector::{HnswParams, create_hnsw_index_with_ops_sql};

let params = HnswParams::new(16, 100).unwrap();
let sql = create_hnsw_index_with_ops_sql("documents", &params, "vector_cosine_ops");
```

Parameters:
| Parameter | Range | Guidance |
|---|---|---|
| `m` | 2-100 | Higher = better recall, more memory. Default: 16 |
| `ef_construct` | 4-1000 | Higher = better index quality, slower build. Default: 100 |

Tune query-time recall with:

```sql
SET hnsw.ef_search = 64;   -- default 40; higher = better recall
```

Generated by `set_hnsw_ef_search_sql(64)`.

### IVF-Flat

Better for very large tables (>1M rows) where HNSW memory overhead is prohibitive.

```rust
use ancora_memory::backends::pgvector::{IvfFlatParams, create_ivfflat_index_sql};

let params = IvfFlatParams::for_table_size(500_000); // sqrt(500k)=707 lists
let sql = create_ivfflat_index_sql("documents", params.lists);
// SET ivfflat.probes before query:
let probe_sql = set_ivfflat_probes_sql(params.probes);
```

The index must be built after rows are loaded (unlike HNSW which can be built
first). Rebuild after bulk loads.

## Distance metrics

| Metric | Operator | `metric_query_sql` key | Use when |
|---|---|---|---|
| Cosine | `<=>` | `"cosine"` | Text embeddings (default) |
| Inner product | `<#>` | `"dot"` | Dot-product tasks |
| L2 / Euclidean | `<->` | `"l2"` | Image embeddings |

```rust
let sql = metric_query_sql("cosine", "documents", 10, 0);
```

## Hybrid search

Combines dense vector ranking with PostgreSQL full-text search:

```
score = alpha * cosine_similarity + (1 - alpha) * ts_rank
```

- `alpha=1.0`: pure vector search
- `alpha=0.0`: pure keyword search
- `alpha=0.7`: balanced (recommended starting point)

The keyword signal uses `to_tsvector(payload->>'text')` against a
`plainto_tsquery`. Store searchable text under the `text` key in the payload,
or use `hybrid_query_column_sql` to specify a different key.

## Metadata filters

Filters translate to JSONB `payload` column conditions:

```rust
use ancora_memory::backends::pgvector::{filter_to_sql, build_where_clause};
use ancora_memory::vector_store::{Filter, PayloadValue};

let f = Filter::Eq("source".to_owned(), PayloadValue::String("arxiv".to_owned()))
    .and(Filter::Gt("year".to_owned(), PayloadValue::Integer(2020)));
let (where_clause, params) = build_where_clause(&f, 1); // 1=embedding is $1
let sql = cosine_query_with_filter_sql("documents", 10, 0, &where_clause.replace("WHERE ", ""));
```

Maximum nesting depth: 16 levels (enforced by `validate_filter_depth`).

## Idempotent batch upsert

Use the journal table to make retried batch upserts safe:

```rust
use ancora_memory::backends::pgvector::{
    create_journal_table_sql, check_journal_sql, insert_journal_sql,
};

// On startup
// execute create_journal_table_sql("upsert_journal")...

// Before each batch
// let already_done = execute check_journal_sql("upsert_journal") with (key,)...
// if already_done.row_count is Some(_): skip

// After each batch
// execute insert_journal_sql("upsert_journal") with (key, collection, count)...
```

Purge old entries periodically:

```rust
use ancora_memory::backends::pgvector::purge_journal_sql;
let sql = purge_journal_sql("upsert_journal");
// execute sql with ("30 days",)...
```

## Schema migrations

Use `pgvector_migration` for schema lifecycle operations:

```rust
use ancora_memory::backends::pgvector_migration::{
    create_schema_version_table_sql, upsert_version_sql, resize_embedding_column_sql,
};
```

Dimension resizing (`resize_embedding_column_sql`) drops and re-adds the
embedding column. Run in a transaction and keep a backup.

## Reconnect policy

| SQLSTATE class | Action |
|---|---|
| `08xxx` (connection errors) | Retry with backoff |
| `40001` / `40P01` (serialization / deadlock) | Retry in new transaction |
| `42xxx` / `23xxx` (syntax / constraint) | Abort -- logic error |

Backoff: `100ms * 2^attempt`, capped at 30 seconds, max 5 retries.

```rust
use ancora_memory::backends::pgvector::{classify_pg_error, retry_delay_ms, ConnectAction};

match classify_pg_error("40001") {
    ConnectAction::RetryTx => { /* begin new transaction */ }
    ConnectAction::Retry   => { /* reconnect after delay */ }
    ConnectAction::Abort   => { /* surface error */ }
}
```

## Integration tests

Integration tests that need a live database are marked `#[ignore]` and read
`TEST_DATABASE_URL` from the environment. Run them with:

```
TEST_DATABASE_URL=postgres://user:pass@localhost/test \
  cargo test -p ancora-memory -- --ignored
```
